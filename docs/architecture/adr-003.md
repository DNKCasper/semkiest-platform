# ADR-003: Async Job Processing with BullMQ and Redis

**Status:** Accepted
**Date:** 2024-01-12
**Deciders:** Platform Engineering Team

---

## Context

The platform needs to run long-duration background tasks that should not block the API request lifecycle. Examples include:

- Executing automated test suites (potentially minutes)
- Sending notification emails or Slack messages
- Exporting reports to S3
- Syncing data with external integrations (Jira, Asana)

These tasks require reliable retry semantics, concurrency control, and observability into job state.

## Decision

We will use **BullMQ** with **Redis** as the job queue. A dedicated **`@semkiest/worker`** process will consume jobs produced by the API.

## Considered Alternatives

### Option A: In-process async execution (setTimeout / setImmediate)
Run background work inside the API process with no queue.

**Rejected because:**
- Work is lost if the API process crashes
- No retry mechanism for failures
- Long-running tasks block API memory and CPU
- No visibility into job status

### Option B: Database-backed job queue (Prisma + polling)
Store jobs in the `JobQueue` table and poll from a worker.

**Partially adopted:** We mirror job state in `JobQueue` for audit/observability, but BullMQ is the actual execution engine.

**Rejected as sole approach because:**
- Polling introduces latency and unnecessary database load
- Implementing retries, backoff, concurrency limits, and priorities from scratch is substantial work

### Option C: AWS SQS or similar managed queue
Use a cloud-managed message queue.

**Rejected because:**
- Adds a cloud dependency to local development
- Higher operational overhead (IAM, VPC, dead-letter queues) for a team of this size
- Redis is already required for BullMQ; SQS would add a second infrastructure dependency

### Option D: BullMQ + Redis (chosen)

**Pros:**
- Battle-tested library with native TypeScript support
- Built-in retry, exponential backoff, rate limiting, and delayed jobs
- Job events enable real-time status updates (WebSockets)
- Redis is fast and already used for other caching/session needs
- BullMQ Board provides a web UI for job monitoring

**Cons:**
- Redis becomes a required infrastructure dependency
- Redis does not persist jobs indefinitely — Redis memory must be sized appropriately
- At-least-once delivery semantics mean workers must be idempotent

## Architecture

```
API (Producer)          Redis (Queue)          Worker (Consumer)
    |                       |                       |
    |-- enqueue job ------->|                       |
    |                       |<-- poll for jobs -----|
    |                       |--- deliver job ------->|
    |                       |                       |-- process
    |                       |<-- acknowledge --------|
    |<-- update DB via API  |                       |
```

- The API enqueues jobs using BullMQ's `Queue` class
- The worker uses BullMQ's `Worker` class to process jobs
- Job state is mirrored to the `JobQueue` database table for long-term audit trails
- `WORKER_CONCURRENCY` controls how many jobs process in parallel per worker instance

## Consequences

- Redis is a required infrastructure service (included in Docker Compose)
- Workers must be **idempotent** — a job may be delivered more than once
- Job payloads must be JSON-serializable
- Long-running jobs should emit progress events to enable real-time status reporting
- `WORKER_QUEUES` can limit a worker instance to specific queue types, enabling horizontal scaling by queue
- Job failures are retried up to `maxAttempts` with exponential backoff; final failures are moved to the dead-letter queue in Redis and recorded in `JobQueue` with `status: "failed"`
