# ADR-001: Monorepo with Turborepo and pnpm Workspaces

**Status:** Accepted
**Date:** 2024-01-10
**Deciders:** Platform Engineering Team

---

## Context

The SemkiEst Platform consists of several closely related services — a Next.js web frontend, an Express API backend, a BullMQ worker process, and shared infrastructure packages (database client, shared types, shared utilities, environment configuration). These services share TypeScript types and configuration, and are developed and deployed by the same team.

The key question: should these services live in a single monorepo or in separate repositories?

## Decision

We will use a **monorepo** managed by **Turborepo** with **pnpm workspaces**.

## Considered Alternatives

### Option A: Separate repositories per service
Each service (api, web, worker) lives in its own Git repository.

**Pros:**
- Independent deployments and release cycles
- Teams can use different tooling per repo
- Granular access control

**Cons:**
- Shared code (types, config) must be published as npm packages or duplicated
- Cross-service changes require PRs across multiple repos
- Local development requires running multiple repos
- Keeping shared types in sync across repos is error-prone

### Option B: Single flat repository (all files at root)
All service code lives in the root directory without workspace separation.

**Pros:**
- Simplest possible structure
- No cross-package dependency management

**Cons:**
- No isolation between services — one service's dependencies pollute all others
- Cannot develop or test services independently
- Does not scale as the codebase grows

### Option C: Monorepo with Turborepo + pnpm workspaces (chosen)

**Pros:**
- Shared code (`@semkiest/shared-types`, `@semkiest/db`) is imported directly without publishing
- TypeScript type changes propagate instantly across all packages
- Turborepo's task graph ensures correct build order and caches unchanged packages
- pnpm's workspace protocol prevents duplicate dependencies
- A single `git clone` provides the full development environment
- CI/CD runs a single pipeline with caching benefits

**Cons:**
- Slightly higher initial tooling complexity
- All services share a single lockfile — dependency upgrades affect all packages simultaneously

## Consequences

- All new services must be added as a workspace package under `apps/` (runnable services) or `packages/` (libraries)
- Shared code lives in `packages/` with a `@semkiest/` namespace
- The `turbo.json` task pipeline must be updated when new build tasks are introduced
- pnpm >= 9 is required; npm and yarn are not supported
- The Turborepo remote cache is recommended for CI to maintain fast build times as the codebase grows
