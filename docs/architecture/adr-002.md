# ADR-002: PostgreSQL with Prisma ORM

**Status:** Accepted
**Date:** 2024-01-10
**Deciders:** Platform Engineering Team

---

## Context

The platform requires a relational database to store projects, test runs, results, users, and job records. We need to choose:

1. Which database engine to use
2. How to interact with it from TypeScript services

## Decision

We will use **PostgreSQL 15** as the primary database and **Prisma ORM** as the data access layer, centralized in the `@semkiest/db` package.

## Considered Alternatives

### Database Engine

| Option | Reason Not Chosen |
|--------|-------------------|
| MySQL / MariaDB | Less powerful JSON support; fewer advanced index types |
| SQLite | Not suitable for multi-process / multi-instance deployments |
| MongoDB | Relational data model with foreign keys fits the domain better |
| CockroachDB | Additional operational complexity without clear benefit at current scale |

PostgreSQL was chosen for its proven reliability, rich feature set (JSONB, arrays, full-text search, `pg_trgm`), and strong Prisma support.

### ORM / Query Layer

| Option | Reason Not Chosen |
|--------|-------------------|
| Raw SQL (`pg` driver) | No type safety; migrations must be maintained manually |
| Knex.js | Query builder, not a full ORM; type safety requires extra tooling |
| TypeORM | Decorator-heavy API; historically slower to adopt new TypeScript features |
| Drizzle ORM | Good option, but Prisma has better migration tooling and wider community support |
| Prisma (chosen) | Type-safe client auto-generated from schema; first-class migration workflow |

## Consequences

### Positive
- The Prisma schema (`packages/db/prisma/schema.prisma`) is the single source of truth for the data model
- Auto-generated TypeScript types eliminate an entire class of runtime errors
- `prisma migrate` provides a reproducible migration history
- Prisma Studio (`npx prisma studio`) provides a GUI for ad-hoc data inspection
- The `@semkiest/db` package is the only consumer of the Prisma client — other packages import from it

### Constraints
- The Prisma client must be regenerated after every schema change: `pnpm --filter @semkiest/db run db:generate`
- `DIRECT_URL` must be set when PgBouncer connection pooling is in use (Prisma requires a direct connection for migrations)
- Prisma does not support automatic rollbacks — rolling back requires a forward migration
- N+1 query problems must be mitigated with `include` / `select` on Prisma queries rather than lazy-loading

### Migration Policy
- Every schema change requires a migration file committed alongside the schema
- Migration names must be descriptive: `add_project_tags`, `create_test_results_table`
- Migrations are never edited after they are merged to `main`
- Production deployments run `prisma migrate deploy` (not `migrate dev`)
